# 🏗️ Архитектура проекта: ESP32 Smart Bridge (RTOS Edition)

## 1. Концепция
Система строится на базе операционной системы реального времени **FreeRTOS** с использованием **Task-Oriented Architecture**. Программный код разделен на функциональные слои для обеспечения модульности, тестируемости и отказоустойчивости.

Ключевые принципы:
*   **Изоляция ресурсов:** Каждая задача владеет своими ресурсами (UART, WiFi) и общается с другими через потокобезопасные примитивы.
*   **Data Consistency:** Использование паттерна "Snapshot" и мьютексов для атомарного чтения/записи состояния.
*   **Separation of Concerns:** Четкое разделение драйверов железа, бизнес-логики и потоков исполнения.

## 2. Файловая структура (Layered Layout)

```text

│   │   ├── inc/                 # SystemState.h, Config.h, EventGroups.h
│   │   └── src/                 # SystemState.cpp
│   ├── Drivers/                 # Слой 0: Низкоуровневые адаптеры (HAL)
│   │   ├── inc/                 # Uart_Drv.h, Fifo_Drv.h, WiFi_Drv.h, Storage_Drv.h
│   │   └── src/                 # Uart_Drv.cpp, Fifo_Drv.cpp, WiFi_Drv.cpp, Storage_Drv.cpp
│   ├── Services/                # Слой 2: Бизнес-логика и Обработка (Logic)
│   │   ├── inc/                 # Protocol_Srv.h, Api_Srv.h, Cmd_Srv.h, Logger_Srv.h
│   │   └── src/                 # Protocol_Srv.cpp, Api_Srv.cpp, Cmd_Srv.cpp, Logger_Srv.cpp
│   └── Tasks/                   # Слой 3: Потоки исполнения (Active Tasks)
│       ├── inc/                 # Uart_Task.h, Network_Task.h, System_Task.h
│       └── src/                 # Uart_Task.cpp, Network_Task.cpp, System_Task.cpp
├── src/
│   └── main.cpp                 # Точка входа: инит слоев и запуск планировщика
├── platformio.ini               # Конфигурация проекта
└── ...
```

## 3. Описание слоев (Layer Responsibilities)

### 3.1. App/Core (Passive Foundation)
*   **SystemState:** "Единственный источник правды". Хранит текущую температуру, уставку, статус реле и время последнего обновления. Все поля защищены мьютексом. Предоставляет метод `GetData()` для получения согласованной копии данных. Парсит данные из `HEALTH_DATA` (0x04) и `USER_DATA` (0x03).
*   **EventGroups:** Определяет битовые флаги событий (например, `BIT_WIFI_CONNECTED`, `BIT_DATA_RECEIVED`), которые позволяют задачам синхронизироваться без активного опроса.

### 3.2. App/Drivers (Hardware Abstraction)
*   **Fifo_Drv:** Универсальный кольцевой буфер (Lock-free), реализованный в ООП стиле. Обеспечивает надежное хранение байтов между аппаратным буфером и логикой парсинга.
*   **Uart_Drv:** Инициализирует `Serial2`. Интегрирует `Fifo_Drv` для фонового приема данных. Предоставляет потокобезопасные методы чтения и записи.
*   **WiFi_Drv:** Отвечает за подключение к точке доступа, обработку событий и переподключение при обрыве.
*   **Storage_Drv:** Инициализирует LittleFS, читает/пишет конфигурационные файлы JSON.

### 3.3. App/Services (Business Logic)
*   **Protocol_Srv:** Знает структуру пакета v1.2. Парсит поток байт из FIFO, проверяет CRC. Формирует байтовые массивы команд. Вызывается из задач.
*   **Api_Srv:** Формирует JSON-ответы для фронтенда на основе данных из `SystemState`.
*   **Cmd_Srv:** Обертка над FreeRTOS Queue. Позволяет любой задаче (например, Web) безопасно положить команду в очередь на отправку.

### 3.4. App/Tasks (Active Execution)
*   **Uart_Task (Core 1, High Priority):**
    1.  **Событийная модель (Reactive):** Задача находится в состоянии ожидания (Blocked), пока в FIFO не поступят данные от Исполнителя (STM32).
    2.  **Обработка Push-пакетов:** При получении пакетов `0x03` (Температура) или `0x04` (Здоровье) мгновенно обновляет `System_State`.
    3.  **Watchdog связи:** Самостоятельно отслеживает интервал между пакетами. Если данные не поступали более 65 секунд, инициирует проверочный `PING (0x10)`.
    4.  **Priority Command Handling:** Мгновенно просыпается при появлении команд в `Cmd_Srv` от пользователя.
*   **Network_Task (Core 0, Medium Priority):**
    *   Поддерживает WiFi соединение.
    *   Запускает `WebServer.h`.
    *   Обрабатывает HTTP-запросы в контексте своих коллбеков.

### 3.5. Presentation Layer (Слой Представления - Web UI)
Этот слой полностью изолирован от системной логики и физически размещен в файловой системе **LittleFS**.
*   **Реализация:** Набор статичных файлов (`index.html`, `style.css`, `app.js`) в папке `/data`.
*   **Взаимодействие:** Общается с системой исключительно через **API Layer** (`Api_Srv` и HTTP эндпоинты). 
*   **Преимущество:** Позволяет обновлять интерфейс пользователя без пересборки и перепрошивки основного микрокода (Firmware), а также масштабировать систему до мобильных приложений, использующих тот же API.

## 4. Потоки данных (Data Flow)

### 4.1. Синхронизация состояния (Push-модель: STM32 -> Bridge)
1.  **STM32 (Исполнитель):** Автономно измеряет параметры и отправляет пакеты `0x03/0x04` с заданным интервалом (например, 30 сек) или по значимому событию (Delta-изменение).
2.  **Бридж (ESP32):** Работает как "слушатель". При получении байт `Uart_Task` просыпается, парсит фрейм и обновляет кэш в `System_State`.
3.  **Диагностика:** Если STM32 молчит дольше критического порога, Бридж переходит в режим активного опроса (Pull) для проверки жизнеспособности канала связи.
4.  **UI:** Браузер забирает данные из кэша ESP32 через JSON API, обеспечивая мгновенный отклик без ожидания транзакции по UART.

### 4.2. Управление (Web -> STM32)
1.  **Browser** шлет POST запрос `/api/control`.
2.  **Network_Task** парсит параметры -> вызывает `Cmd_Srv::Enqueue(CMD_SET_TEMP, 25.0)`.
3.  `Cmd_Srv` кладет команду в `xQueue`.
4.  **Uart_Task** в своем цикле видит данные в очереди.
5.  Вызывает `Protocol_Srv::PackFrame()` -> получает байты.
6.  Вызывает `Uart_Drv::SendBytes()` -> байты уходят на STM32.

## 6. Технические особенности реализации (C++ / OOP)

### 6.1. Оптимизация FIFO (Bitwise Masking)
Для обеспечения максимальной скорости работы кольцевого буфера используется побитовое И (`&`) вместо деления по модулю (`%`). 
*   **Условие:** Размер буфера (`SIZE`) обязан быть степенью двойки.
*   **Логика:** Операция `(index + 1) & (SIZE - 1)` автоматически сбрасывает индекс в 0 при достижении границы `SIZE`.

### 6.2. Шаблоны (C++ Templates)
Класс `Fifo_Drv` реализован как шаблон (`template <uint16_t SIZE>`).
*   **Определение:** Сообщает компилятору, что `SIZE` является параметром времени компиляции. Это «чертеж» для создания реального класса.
*   **Преимущества:** Статическая типизация и выделение памяти внутри объекта без `malloc`.
*   **Header-only:** Код шаблонов находится в `.h` файле, так как компилятор генерирует код в момент объявления.

### 6.3. Безопасность компиляции (Static Assert)
Используется `static_assert((SIZE & (SIZE - 1)) == 0, "...")`. Это останавливает сборку, если размер буфера не является степенью двойки.

### 6.4. Списки инициализации конструктора
Синтаксис `: _var(value)` инициализирует переменные в момент выделения памяти, что эффективнее присваивания в теле конструктора.

### 6.5. Статические члены класса (Static Members)
Используется для ресурсов в единственном экземпляре (например, `_rxFifo`). Объявляется в `.h`, определяется в `.cpp`. Гарантирует единственность буфера для физического порта.

### 6.6. Инкапсуляция и защита данных (Private Access)
Внутренний буфер `_rxFifo` помещен в `private`. Доступ возможен только через публичные методы `Update()` и `ReadByte()`.

### 6.7. Композиция против Наследования (Has-A vs Is-A)
Используется принцип **Композиции**: `Uart_Drv` **имеет** внутри себя `Fifo_Drv`. Это делает систему модульной и гибкой.

### 6.8. Паттерн Singleton (Синглтон Мейерса)
Для управления глобальным состоянием системы используется паттерн **Singleton**.
*   **Функция:** `static System_State& instance()`.
*   **Роль `static`:** Позволяет любой задаче (UART, Web) получить доступ к данным через `System_State::instance()`, не создавая объект вручную.
*   **Роль `&` (Ссылка):** Гарантирует, что функция возвращает **оригинал** объекта из памяти, а не его копию. Любое изменение данных (например, температуры) через эту ссылку мгновенно становится видимым для всех остальных задач.
*   **Lazy Initialization:** Объект создается в памяти ESP32 ровно в тот момент, когда к нему произошло первое обращение. Это исключает ошибки при старте системы, когда один драйвер пытается обратиться к другому, еще не инициализированному.
*   **Безопасность:** Конструктор класса помещен в `private`, что делает невозможным случайное создание «дубликатов» состояния в других частях программы.

#### Пример прохождения (Жизненный цикл):
1.  **Вызов:** Задача `Uart_Task` впервые пишет: `System_State::instance().UpdateUserData(...)`.
2.  **Инициализация (Lazy Init):** Внутри функции `instance()` процессор видит строчку `static System_State inst;`.
    *   Поскольку это первый вызов, в памяти RAM выделяется место.
    *   **Автоматически** вызывается **Конструктор** `System_State()`.
3.  **Подготовка (Constructor):** Внутри конструктора создается Мьютекс (`xSemaphoreCreateMutex`) and обнуляется память (`memset`). Это гарантирует, что "замок" для защиты данных будет готов **до** того, как к данным прикоснутся.
4.  **Выдача Оригинала:** Функция возвращает ссылку (`&`) на этот свежесозданный объект.
5.  **Действие:** Выполняется метод `.UpdateUserData(...)`. Он захватывает созданный в п.3 мьютекс, безопасно меняет температуру и открывает мьютекс.
6.  **Повторный доступ:** Когда через секунду Web-сервер вызовет `System_State::instance().GetData()`, пункты 2 и 3 будут пропущены. Система сразу выдаст ссылку на тот же самый "живой" оригинал объекта в памяти.

### 6.9. Статические методы и аппаратные ресурсы (Hardware-bound Static Classes)
Драйверы физических устройств (UART, WiFi) реализованы как классы со статическими методами.
*   **Связь с железом:** Физический ресурс (например, Serial2) уникален для чипа. Использование статических методов исключает бессмысленное создание множества объектов («экземпляров») для одного и того же порта.
*   **Физика памяти:** Статические переменные (члены класса) закрепляются за постоянными адресами в RAM. Статические методы знают эти адреса и работают с ними напрямую.
*   **Отсутствие `this`:** В отличие от обычных методов, статические методы не получают скрытый указатель на экземпляр (`this`). Это делает вызов функции быстрее и экономнее по стеку, так как процессору не нужно выяснять, «чьи» данные он сейчас обрабатывает — данные всегда общие и закреплены за классом.
*   **Синтаксис вызова:** Используется оператор разрешения области видимости `::` (например, `Uart_Drv::SendBytes`). Это явно указывает, что мы обращаемся к общему сервису класса, а не к его конкретному экземпляру через точку.
*   **Принцип "Нет объекта — нет конструктора":** Поскольку аппаратный ресурс (например, чип WiFi) в системе один, нам не нужно создавать его "копии" (экземпляры/объекты) в памяти. Конструктор в C++ нужен для инициализации конкретного объекта. Если мы не создаем объекты — конструктор не требуется.
*   **Безопасность (Advanced):** Чтобы исключить случайное создание объекта статического класса (например, `WiFi_Drv myWifi;`), конструктор по умолчанию явно удаляется в коде драйвера (`WiFi_Drv() = delete;`). Это гарантирует, что драйвер всегда будет существовать в единственном числе как системный сервис.

#### 🛠️ Архитектурные принципы выбора: Статический сервис vs Динамический объект
В проекте SmartHeater выбор между статическим классом и созданием объекта диктуется следующими правилами промышленного стандарта:

1.  **Принцип Единственности (Singularity):**
    *   **Статический сервис:** Если физический ресурс (WiFi, UART, Flash) уникален для чипа. Создание объектов здесь опасно, так как они могут конкурировать за управление одним и тем же железом.
    *   **Объект:** Если сущностей может быть много (например, `Fifo_Drv` — один для приема, другой для логов).
2.  **Принцип Состояния (Statefulness):**
    *   **Статический сервис:** Идеален для "чистых функций" или "переводчиков" (например, `Api_Srv`). Он выполняет действие, не сохраняя "характер" или сложный контекст между вызовами.
    *   **Объект:** Нужен там, где важен контекст (например, если бы у нас было 10 одинаковых датчиков температуры, каждый был бы отдельным объектом со своим ID и калибровкой).
3.  **Принцип Детерминизма Памяти (Memory Safety):**
    *   **Статический сервис:** Размещается в памяти один раз при старте в фиксированном месте. Это исключает **фрагментацию кучи (Heap Fragmentation)** — главную причину зависаний контроллеров при долгой работе. В Advanced-разработке приоритет всегда отдается статическому размещению инфраструктуры.
4.  **Принцип Эффективности (Stack Efficiency):**
    *   Статические методы не передают скрытый указатель `this`. Это экономит место в стеке и такты процессора, что критично для высоконагруженных задач (UART, прерывания).
